# 평가문항
## 항목 1
- SSH 포트가 20022로 변경되었고, Root 원격 접속이 차단되었는가?
  - 방법: `/etc/ssh/sshd_config`에서 `Port 20022`로 설정하고 `PermitRootLogin no`로 변경한 뒤 `systemctl restart sshd` 또는 `service sshd restart` 실행. 방화벽 규칙에서 22/tcp를 차단하고 20022/tcp를 허용.
  - 이유: 포트 변경과 Root 접속 차단은 자동 스캐닝·무차별 대입 공격 노출을 줄이고, 루트 계정으로 직접 접근할 수 없게 하여 권한 상승 위험을 낮추기 위함.
- 방화벽이 활성화되어 있고(택1: UFW 또는 firewalld), 20022/tcp와 15034/tcp만 허용되는가? (실제 구현: UFW 활성화)
  - 방법: UFW 사용(현재 구현)에 대해 `ufw enable`, `ufw allow 20022/tcp`, `ufw allow 15034/tcp`, `ufw default deny incoming`를 실행했습니다. (firewalld 대신 UFW 선택)
  - 이유: 최소 권한 네트워크 접근 원칙으로 불필요한 포트를 차단해 공격면(attack surface)을 줄임.
- agent-admin/dev/test 계정과 agent-common/core 그룹이 요구사항대로 구성되어 있는가?
  - 방법: `groupadd agent-common` 후 `useradd -m -G agent-common agent-dev` 등으로 계정/그룹 생성, 필요시 `usermod -aG`로 추가. 홈 디렉터리·쉘·패스워드 정책 확인.
  - 이유: 역할 분리와 최소 권한 적용을 위해 계정과 그룹을 명확히 분리하면 권한 관리와 감사가 쉬워짐.
- 앱이 Boot Sequence 5단계 [OK]를 통과하고 "Agent READY"가 출력되는가?
  - 방법: 서비스 초기화 스크립트나 systemd 유닛에서 부팅 단계별 상태를 출력하도록 구현(`logger` 또는 stdout에 단계별 메시지 기록). 부팅 시 `systemctl status` 또는 로그를 확인해 "Agent READY" 표시 확인.
  - 이유: 초기화가 정상적으로 끝났음을 확인하면 외부 서비스(로드밸런서, 오케스트레이터)가 안전하게 트래픽을 보낼 수 있음.
- monitor.sh가 프로세스/포트 상태를 점검하고, 비정상 상태에서 exit 1로 종료되는가?
  - 방법: `pgrep`/`ps`로 프로세스 확인하고 `ss`/`ss -ltnp` 또는 `ss -tlnp`로 포트 바인딩 확인. 이상시 `echo`로 로그 남기고 `exit 1` 반환.
  - 이유: 모니터가 비정상(다운/포트미바인드) 상태를 명확히 알리면 재시작 스크립트나 오케스트레이션 툴이 자동 복구 가능.
- `/var/log/agent-app/monitor.log`가 지정 포맷으로 누적 기록되는가?
  - 방법: `monitor.sh`에서 `date -u '+%Y-%m-%dT%H:%M:%SZ'`와 key=value 형태(예: `time=... status=OK pid=... cpu=... mem=...`)로 `>> /var/log/agent-app/monitor.log`로 누적.
  - 이유: 타임스탬프와 키=값 포맷은 파싱/검색/집계에 유리하고, 감사 증적을 제공함.
- cron 매분 실행으로 monitor.log가 자동 증가하는가?
  - 방법: 크론에 `* * * * * /path/to/monitor.sh >> /var/log/agent-app/monitor.log 2>&1` 항목 추가(또는 `agent-admin` 계정의 crontab 사용).
  - 이유: 주기적 상태 수집으로 추세 분석 및 신속한 이상 탐지가 가능.
- monitorlog 용량 관리(10MB/10개)가 설정되어 있고 동작을 설명할 수 있는가?
  - 방법: `logrotate` 설정 파일에 `size 10M`, `rotate 10`, `compress`, `missingok`, `copytruncate` 등을 추가하거나, 스크립트로 `if [ $(du -m file | cut -f1) -ge 10 ]` 형식으로 회전 및 보관(최대 10개) 구현.
  - 이유: 로그가 무한히 늘어나 디스크를 채우는 것을 방지하고, 필요한 만큼의 과거 기록을 보관하기 위함.
## 항목 2
- monitor.sh에서 프로세스 식별(pgrep/ps 등)과 포트 확인(ss/netstat 등)에 사용한 명령과 선택 이유를 설명할 수 있는가?
  - 설명: `pgrep -f`는 프로세스 이름/커맨드라인을 간단히 확인할 때 안전하고 간결함. 복잡한 매칭이 필요하면 `ps aux | grep '[p]attern'`를 사용. 포트 확인은 `ss -tlnp`를 권장(더 빠르고 netstat보다 최신). `lsof -i :PORT`는 어떤 PID가 포트를 점유했는지 정확히 알려줌. 선택 기준은 성능, 설치 여부, 정확성(프로세스 식별과 포트 바인딩 매핑 가능 여부).
- CPUMEMDISK 값을 어떤 방식으로 추출/파싱했고, 로그 포맷을 왜 그 형태로 고정했는지 설명할 수 있는가?
  - 설명: CPU는 `/proc/stat` 또는 `top -b -n1`의 요약을 `awk`로 파싱하여 사용률을 계산하고, 메모리는 `/proc/meminfo`의 MemTotal/MemAvailable을 이용해 사용률을 계산. 디스크는 `df --output=source,pcent /path` 또는 `df -m`을 `awk`로 파싱. 로그 포맷을 `time=... cpu=... mem=... disk=... status=...`처럼 고정하면 스크립트/Prometheus 텍스트 파서/ELK 등에서 일관되게 파싱 가능하므로 자동화와 알람 규칙 적용이 쉬워짐.
- 소유자(agent-dev)와 실행자(agent-admin, cron) 권한 정책을 어떻게 만족시켰는지(소유/그룹/권한) 설명할 수 있는가?
  - 설명: 민감 파일(예: `apikeys`)은 `chown agent-dev:agent-core apikeys`와 `chmod 640 apikeys`로 소유자를 `agent-dev`, 그룹을 `agent-core`로 제한하고 읽기 권한을 소유자와 그룹으로만 허용. 스크립트/서비스 실행은 `agent-admin` 또는 시스템 서비스 유닛에서 `User=agent-admin`로 실행하여 운영자가 관리하도록 함. cron 작업은 `agent-admin`의 crontab에 등록하거나 root crontab에서 `su - agent-admin -c '/path/monitor.sh'`로 실행해 실행 주체를 명확히 함.
- 용량 기반 로그 관리(10MB/10개)를 어떤 방식(logrotate/스크립트)으로 구현했는지 설명할 수 있는가?
  - 설명: 표준 방식은 `logrotate` 사용: 예시 설정에 `size 10M`, `rotate 10`, `create 640 agent-dev agent-core`, `compress`를 사용하면 파일이 10MB 이상일 때 회전하고 최대 10개 보관. 대체 방식은 스크립트로 파일 크기 체크 후 `mv file file.1 && gzip file.1` 하고 오래된 인덱스 삭제하는 방식이며, `logrotate`가 더 표준적이고 신뢰성 높음.
## 항목 3
- SSH 포트 변경과 Root 접속 차단이 왜 보안에 효과적인지 위협 모델 관점에서 설명할 수 있는가?
  - 설명: 자동 스캐너와 스크립트 기반 공격은 기본 포트(22)에 집중되므로 포트를 변경하면 무차별 스캔으로 인한 노출을 줄임. 루트 원격 로그인을 금지하면 공격자가 루트로 직접 접근해 즉시 전체 시스템을 장악하는 경로를 차단하므로 권한 상승을 위한 추가 취약점(예: sudo 권한 취득) 없이는 피해 범위를 줄일 수 있음. 다만 이는 단독 방어책이 아니라 브루트포스 차단, 패치 관리, 강력한 인증(키 기반)과 함께 사용되어야 함.
- apikeys와 로그 디렉토리를 agent-core로 제한한 이유를 최소 권한 원칙"으로 설명할 수 있는가?
  - 설명: 최소 권한 원칙(Least Privilege)은 각 계정/프로세스가 수행에 필요한 권한만 갖도록 제한하는 원칙임. `apikeys`와 중요 로그를 `agent-core`만 접근 가능하게 하면 계정 탈취 시 노출 범위를 줄이고, 내부 오용 및 정보 유출 위험을 감소시킴. 또한 감사와 책임 추적이 쉬워짐.
- "경고는 출력하되 종료하지 않는 항목 (방화벽 비활성/임계치 초과)을 분리한 운영상의 이유를 설명할 수 있는가?
  - 설명: 일부 상태(예: 방화벽 비활성, 임계치 초과)는 즉시 서비스 종료로 이어지면 가용성에 큰 영향을 줄 수 있음. 운영상 중요한 차이점은 '치명적(서비스 불가)'과 '주의 필요(운영자 개입)'로 구분하는 것. 경고만 발생시키면 자동화된 재시작 루틴에 의해 불필요하게 서비스가 반복 재기동되는 것을 막고, 운영자가 원인 분석 후 수동 개입으로 안정적으로 대응할 수 있음.
- 리다이렉션 기호>와 차이를 설명하고, 로그 누적에 2> 가 필요한 이유를 설명할 수 있는가?
  - 설명: `>`는 표준출력(stdout)을 지정한 파일로 쓰고(기존 내용을 덮어씀), `>>`는 이어쓰기(append). `2>`는 표준에러(stderr)를 지정된 파일로 리다이렉트. 로그 누적 시에는 `>> /path/log 2>&1`처럼 stdout과 stderr를 모두 누적해야 오류 메시지도 로그에 포함되어 문제 원인 분석이 가능함. 단순히 `>`를 사용하면 이전 로그가 삭제되므로 위험함.
## 항목 4
- 모니터링 대상이 웹 서버(Nginx 등)로 바뀐다면, monitor.sh에서 바꿔야 할 핵심 포인트(프로세스/포트/로그/임계값)를 설명할 수 있는가?
  - 설명: (1) 프로세스명(예: `nginx`) 확인 로직으로 변경, (2) 포트 체크를 80/443(또는 서비스가 바인딩한 포트)로 변경, (3) 접근성 체크를 HTTP 요청(예: `curl -s -o /dev/null -w "%{http_code}" http://localhost/health`) 같은 애플리케이션 레벨 헬스체크로 보완, (4) 로그 경로(`/var/log/nginx/access.log`, `error.log`)와 로그 파싱 규칙 조정, (5) 임계값(초당 요청 수, 응답시간, 에러율) 재설정.
- "프로세스는 살아있는데 포트가 안 열리는 상황"을 발견했다면, 원인 후보와 확인 순서를 설명할 수 있는가?
  - 설명: 원인 후보: (a) 서비스가 localhost로만 바인딩(외부 미노출), (b) 방화벽 또는 iptables에 의해 차단, (c) 포트 충돌(다른 프로세스가 먼저 바인딩), (d) 서비스 내부 오류로 리스닝 실패(설정오류), (e) 소켓 리소스 부족. 확인 순서: `ss -ltnp`로 리스닝 확인 → `ps`/`pgrep`로 프로세스 PID 확인 → `lsof -p PID`로 열린 소켓 확인 → 서비스 로그(예: `journalctl -u service` 또는 error.log) 확인 → 방화벽 규칙 확인 → 설정파일(바인드 주소/포트) 검토.
  - 추가(strace 활용): 로그나 설정 파일로 원인이 파악되지 않을 때, 프로세스가 커널과 주고받는 시스템 콜을 직접 추적하기 위해 `strace`를 활용합니다. 예를 들어 `strace -p <PID> -e trace=network,file` 또는 기동 시 `strace -e trace=bind,listen` 명령을 통해 `bind()` 시스템 콜에서 `EADDRINUSE`(이미 사용 중) 에러가 떨어지는지, 권한 부족으로 `EACCES` 에러가 발생하는지 커널 레벨에서 명확한 원인을 식별할 수 있습니다.
- 로그가 급증해 디스크가 가득 찰 위험이 있다면, 운영자가 취할 대응(단기/중기)을 설명할 수 있는가?
  - 설명: 단기: 불필요한 로그 레벨을 낮추거나(예: debug→info), 로그 회전 즉시 수행(`logrotate -f`), 오래된 로그 압축·삭제, 임시로 디스크 정리(캐시/임시파일 삭제), 서비스 일부 일시 중단으로 로그 생성 원인 억제. 중기: 중앙집중형 로그 수집(ELK/Fluentd), 로그 샘플링/레이트리밋 적용, 근본 원인(에러 루프/봇 트래픽) 해결, 로그 정책(보존기간/회전규칙) 재설계, 알람 설정 강화.
